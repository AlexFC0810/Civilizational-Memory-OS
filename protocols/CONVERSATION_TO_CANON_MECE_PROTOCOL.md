# Conversation-to-Canon MECE Protocol

**Purpose:** Convert long, high-insight conversations into durable repository infrastructure without losing breakthroughs, duplicating ideas, canonizing speculation, or leaving execution gaps.

## Governing principle

A conversation is not complete when it produces a strong answer. It is complete when its durable value has been separated, classified, linked, and assigned a future state.

> Preserve the breakthrough. Separate the claims. Expose the uncertainty. Create the next work unit.

## MECE extraction architecture

Every conversation should be decomposed into seven mutually exclusive artifact classes.

### 1. Canonical frameworks

Durable reasoning structures that remain useful beyond the original topic.

Examples:
- blueprint–architecture–implementation;
- succession stress testing;
- capacity–direction–capture–correction;
- historical calibration plus moral transcendence.

### 2. Claim ledger

Atomic factual propositions requiring evidence, verdicts, confidence, and source trails.

Each claim must be tagged:
- established;
- probable;
- contested;
- unsupported;
- category error;
- unresolved.

### 3. Research hypotheses

Explanatory ideas worth testing but not yet canonizing as fact.

Each hypothesis requires:
- mechanism;
- comparison class;
- confounders;
- falsifiers;
- evidence that would raise or lower confidence.

### 4. Research packets and backlog

Bounded cells of uncertainty that an agent or human can execute.

Each packet must contain:
- one exact question;
- one source hierarchy;
- one comparison class;
- one output schema;
- one breaker-evidence requirement.

### 5. Deployment assets

Truthful public compressions:
- essays;
- claim cards;
- debate responses;
- load-bearing questions;
- diagrams;
- datasets;
- AI evals;
- educational sequences.

No deployment asset may exceed the accepted evidence.

### 6. Operating protocols and prompts

Reusable methods that improve future work:
- research prompts;
- red-team procedures;
- integration gates;
- end-of-thread logging routines;
- source-layer protocols.

### 7. Decisions, corrections, and retired language

Record:
- claims adopted;
- claims softened;
- claims rejected;
- terminology retired;
- open disagreements;
- why a previous formulation changed.

This prevents brittle language from returning later.

## Completeness matrix

Before closing a high-insight conversation, inspect every row:

| Dimension | Required output |
|---|---|
| Breakthrough | What genuinely new model emerged? |
| Framework | What reusable structure should enter canon? |
| Claims | What factual propositions appeared? |
| Source layers | Scripture, report, law, practice, empire, modern claim? |
| Countercase | What is the strongest opposing model? |
| Corrections | What did we overstate or misunderstand? |
| Unknowns | What remains unresolved? |
| Falsifiers | What could overturn the working view? |
| Dependencies | Which question must be answered before another? |
| Research | What bounded packets should be created? |
| Deployment | What can safely be published now? |
| Links | Which existing artifacts must be updated or referenced? |
| Ownership | Founder, architect, scout, red team, integrator, editor? |
| Next action | What is the highest-leverage executable move? |

## Canonical status ladder

Do not flatten everything into “knowledge.”

1. **Raw signal** — screenshot, intuition, conversational claim.
2. **Research seed** — structured but not source-closed.
3. **Working hypothesis** — mechanism and falsifier defined.
4. **Provisional verdict** — evidence reviewed but not fully red-teamed.
5. **Accepted framework/claim** — passed integration gate.
6. **Canonical** — durable and repository-defining.
7. **Superseded/retired** — preserved with reason for revision.

## End-of-thread workflow

### Step 1 — Capture

Preserve the full synthesis and conversation context in a dated log.

### Step 2 — Atomize

Split:
- factual claims;
- causal claims;
- moral judgments;
- definitions;
- rhetorical labels;
- strategic insights;
- research questions.

### Step 3 — Deduplicate

Search the repository before creating a new framework. Prefer:
- updating an existing canonical artifact;
- linking a new case to an existing framework;
- creating a new artifact only when the distinction is genuinely new.

### Step 4 — Classify

Route every item into exactly one primary artifact class, with cross-links where necessary.

### Step 5 — Adversarial audit

Ask:
- What is our strongest self-serving interpretation?
- What does the opposing side see correctly?
- Where are we using context as absolution?
- Where are they using modern disgust as analysis?
- What would embarrass us if a specialist reviewed this?

### Step 6 — Create work units

Turn unknowns into bounded research packets, not broad topics.

### Step 7 — Integrate

Update indexes, terrain maps, parent issues, and related frameworks.

### Step 8 — Compress

Create public language only after separating:
- norm;
- authority;
- feasibility;
- implementation;
- enforcement;
- succession;
- correction.

### Step 9 — Close provisionally

Record:
- what is now clearer;
- what is still open;
- what runs next;
- what language is prohibited pending research.

## Anti-loss rules

- No high-insight thread remains only in chat.
- No major framework remains buried inside a conversation log.
- No factual controversy enters canon without a source ledger.
- No hypothesis is stated as a conclusion.
- No research issue is created without an integration target.
- No public answer is stronger than the evidence.
- No previous conclusion disappears silently; supersession is versioned.
- No opponent’s strongest valid point is omitted from the archive.

## Anti-bloat rules

Comprehensiveness is not duplication.

- One canonical home per framework.
- Conversation logs preserve provenance, not replace canon.
- Issues contain execution, not full theory.
- Source ledgers contain evidence, not public rhetoric.
- Public assets contain compression, not the entire research tree.
- Indexes point; they do not restate every artifact.

## Definition of done

A conversation has been fully integrated when:

- every novel framework has a canonical home;
- every important factual claim has a ledger or packet;
- every uncertainty has a status and falsifier;
- every correction is recorded;
- every next action has an owner and dependency;
- every new artifact links back to its source conversation;
- repository navigation has been updated;
- no high-value insight exists only in prose memory.

## Canonical rule

> Conversation is the laboratory. GitHub is the institutional memory. Canon is earned through structure, evidence, red-teaming, and revision.
